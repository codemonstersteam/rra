# Rational Repository Auditor (RRA) — концепция

## Источники
Концепция опирается на цикл статей по рациональной разработке:
- https://codemonsters.team/blog/2026/05/02/rational-design-skills/
- https://codemonsters.team/blog/2026/05/01/rational-design-discipline/
- https://codemonsters.team/blog/2026/04/26/component-tests-skill-passkey-demo/
- https://codemonsters.team/blog/2026/04/07/documentation-as-product-jtbd/
- https://codemonsters.team/blog/2026/04/25/testing-mythology-component-tests/

Используются формулировки автора без перевода: «программа = дерево модулей», «бизнес-логика отделена от I/O», «документация = продукт с 4 JTBD-потребителями», «качество достигается проектированием, а не тестированием».

## Цель и позиционирование
RRA измеряет соответствие репозитория рациональной дисциплине. Не «линтер стиля» и не «отчёт о покрытии тестами».

Два основных критерия:

1. **I/O отделён от бизнес-логики** — структурная корректность по построению.
2. **Документация понятна и проста** — оценивается через JTBD четырёх потребителей.

Прочее (TBD, формула юнит-тестов, vertical slices) — вторичные метрики, не блокирующие итоговое решение.

## Поддерживаемые языки
Go, Java, Python, TypeScript. Каждый поддерживается отдельным «языковым модулем» с собственным каталогом I/O-маркеров и парсером AST. Один язык — одна итерация поставки.

## Архитектура самого RRA — по тем же принципам

Чтобы продукт был «правдив сам себе»:

- **Slice-и (вертикальные срезы)**:
  - `audit-repo` — точка входа CLI/HTTP
  - `audit-iograph` — детектор I/O-границы
  - `audit-docs` — модуль оценки документации (обособлен, можно запускать самостоятельно)
  - `audit-report` — сборка отчёта
- **I/O-модули**: `RepoStore` (git, ФС), `LLMClient` (Anthropic API), `ReportSink`. Никаких сырых `*http.Client` / `os.*` в логике.
- **Бизнес-логика**: чистые функции над AST и текстом. Юнит-тесты без моков.
- **Головной модуль**: оркеструет цепочку, принимает все решения.

## Критерий №1 — отделение I/O от бизнес-логики

Три последовательных прохода. Каждый следующий дороже, но точнее.

### Проход A — статический, по AST и графу импортов (без ИИ)

Решает 70–80% случаев бесплатно.

#### Каталог I/O-маркеров по языкам

**Go**
- Сеть/HTTP: `net/http`, `net`, `google.golang.org/grpc`, `google.golang.org/protobuf`
- БД: `database/sql`, `gorm.io/*`, `*mongo*`, `github.com/jackc/pgx`, `github.com/jmoiron/sqlx`
- Кэш/брокеры: `github.com/redis/*`, `github.com/segmentio/kafka-go`, `github.com/streadway/amqp`, `github.com/nats-io/*`
- Файловая система: `os`, `io`, `bufio`, `path/filepath`
- Недетерминизм: `time.Now`, `time.Since`, `math/rand`, `crypto/rand`
- Сырые типы в сигнатурах: `*sql.DB`, `*http.Client`, `*http.Request`, `*os.File`, `io.Reader`/`io.Writer` как параметр доменной функции

**Java**
- Сеть/HTTP: `java.net.*`, `java.net.http.*`, `org.springframework.web.*`, `okhttp3.*`, `org.apache.http.*`, `feign.*`
- БД: `java.sql.*`, `javax.sql.*`, `jakarta.persistence.*`, `org.hibernate.*`, `org.springframework.data.*`, `org.jdbi.*`
- Кэш/брокеры: `redis.clients.jedis.*`, `io.lettuce.*`, `org.apache.kafka.*`, `com.rabbitmq.*`, `org.springframework.amqp.*`
- ФС: `java.io.File`, `java.nio.file.*`, `java.io.InputStream`/`OutputStream` как параметр доменного метода
- Недетерминизм: `java.time.Clock` (если не инжектится), `java.time.LocalDateTime.now()`, `java.util.Random`, `Math.random`, `UUID.randomUUID` внутри логики
- Сырые типы в сигнатурах: `javax.sql.DataSource`, `EntityManager`, `JdbcTemplate`, `RestTemplate`, `WebClient`
- Аннотационные сигналы нарушения: `@RestController`, `@Repository`, `@Service` со встроенной БД-логикой; `@Autowired` сырого DataSource в доменный сервис

**Python**
- Сеть/HTTP: `requests`, `httpx`, `aiohttp`, `urllib`, `urllib3`, `grpc`
- БД: `psycopg`, `psycopg2`, `sqlalchemy`, `asyncpg`, `pymongo`, `mysqlclient`, `pymysql`, `sqlite3`
- Кэш/брокеры: `redis`, `aioredis`, `kafka`, `confluent_kafka`, `pika`, `aio_pika`, `nats`
- ФС: `open()`, `pathlib.Path.read_*`/`write_*`, `os.*`, `shutil.*`, `io.*`
- Недетерминизм: `datetime.now`, `datetime.utcnow`, `time.time`, `random.*`, `secrets.*`, `uuid.uuid4` внутри логики
- Сырые типы в сигнатурах: `Connection`, `Engine`, `Session`, `httpx.Client`, `requests.Session`

**TypeScript**
- Сеть/HTTP: `fetch`, `axios`, `node-fetch`, `got`, `undici`, `@nestjs/axios`
- БД: `pg`, `mysql2`, `mongodb`, `mongoose`, `typeorm`, `prisma`, `knex`, `drizzle-orm`
- Кэш/брокеры: `ioredis`, `redis`, `kafkajs`, `amqplib`, `nats`, `bullmq`
- ФС: `fs`, `node:fs`, `node:fs/promises`, `node:path`
- Недетерминизм: `Date.now`, `new Date()`, `Math.random`, `crypto.randomUUID` внутри логики
- Сырые типы в сигнатурах: `Pool`, `Client` (из `pg`), `PrismaClient`, `AxiosInstance`, `MongoClient`
- DOM/браузерные: `window`, `document`, `localStorage`, `fetch` напрямую в доменных функциях

#### Правила-блокеры (общие для всех языков)

1. Модуль, классифицированный как «логика», **не должен** иметь транзитивных импортов из I/O-каталога.
2. В контрактах модулей логики **запрещены**:
   - сырые типы клиентов (`*sql.DB`, `DataSource`, `Session`, `Pool`, …);
   - больше одной data-структуры на входе (правило «один data-аргумент на модуль»);
   - возврат через исключение/panic вместо `Result<T, E>` (`(T, error)` в Go, `Either`/`Result` в TS, `Result`/sealed-class в Java, кортеж/`Result`-like в Python).
3. Скрытое состояние: глобальные переменные, синглтоны, `package init`, `static` с состоянием, мутабельные модульные переменные.
4. Юнит-тесты модулей логики, использующие mock-фреймворки (`gomock`, `Mockito`, `unittest.mock`, `jest.mock` с заменой зависимостей), — сигнал, что зависимости втащены внутрь логики. По автору моков быть не должно.

#### Каталог разрешённых импортов в логике

Чтобы статика не давала ложноположительных, ведётся явный список «чистых» библиотек по языкам:

- Go: `errors`, `strings`, `strconv`, `sort`, `math`, `fmt` без `Print*`, доменные пакеты проекта.
- Java: `java.lang.*` (кроме `System.out`/`err`), `java.util.*` (immutable-коллекции и `Optional`), `java.math.*`, доменные пакеты.
- Python: `dataclasses`, `typing`, `enum`, `functools`, `itertools`, `decimal`, `re`, доменные пакеты.
- TypeScript: чистые утилиты (`zod` для валидации, `neverthrow` для `Result`, lodash-fp), доменные модули.

Выход прохода A: классификация модулей + список «жёстких» нарушений с `file:line`.

### Проход B — семантический, через LLM

Только для модулей, которые статика не отнесла однозначно, или для модулей со скрытой зависимостью (интерфейс с неизвестной реализацией).

Промпт-чеклист с обязательной цитатой строки:

```
Дано: код модуля + его публичный контракт.
Ответь Y/N с цитатой строки:
1. Функция модуля формулируется одной фразой? Если да — какой?
2. Контракт: ровно одна data-структура на входе?
3. Возврат — Result<T,E> или эквивалент, без исключений вверх?
4. Внутри модуля есть: сеть, БД, ФС, брокер, время, случайность,
   глобальное состояние? Перечисли вхождения.
5. Все зависимости — автономные I/O-объекты (Store/Client/Publisher),
   а не сырые клиенты?
6. Модуль возвращает управление вызвавшему (нет «прыжков в сторону»)?
```

LLM не имеет права судить «по ощущению» — каждый ответ привязан к цитате. Несоответствие хотя бы одному пункту = нарушение.

### Проход C — сверка с проектной документацией

Если в репозитории есть `docs/contracts-graph.md`, `messages.md`, `intent.md`:

- LLM строит фактический граф вызовов из кода и сверяет с задекларированным.
- Расхождения — отдельный класс нарушений: «код разошёлся со спецификацией».
- По дисциплине автора это означает «остановиться и сообщить» — RRA именно это и делает.

### Итоговая оценка по критерию №1

Структурированный результат, не одна цифра:

- **Жёсткие нарушения** (`*sql.DB` в сигнатуре доменной функции, `@Repository` в slice-логике, `Date.now()` внутри чистой функции) — блокирующие, репозиторий не проходит.
- **Мягкие** — список с локациями.
- **Покрытие графом** — какая доля модулей классифицирована однозначно.

## Критерий №2 — модуль оценки документации (обособленный)

Отдельный slice — можно запускать поверх любого репозитория без I/O-аудита. Это согласуется с принципом «документация = продукт».

Оценка слоистая. ИИ привлекается **только на верхнем слое** и **только если** все нижележащие пройдены. Принцип: «дешёвая модель применяет, дорогая думает» — статика и линтеры дают 90% сигнала бесплатно, LLM добивает остаток.

| Слой | Отвечает на вопрос | Инструмент | ИИ |
|---|---|---|---|
| L1. Читаемость | «написано ли понятно?» | Flesch-Kincaid, Оборнева | нет |
| L2. Стиль | «нет ли когнитивного шума?» | Vale, markdownlint | нет |
| L3. Структурная полнота | «есть ли обязательные файлы и связи?» | shell/Go-скрипт | нет |
| L4. JTBD-присутствие | «есть ли разделы под job каждого потребителя?» | статический чеклист | нет |
| L5. JTBD-пригодность | «закроет ли содержимое job за бюджет?» | LLM-симуляция 4 ролей | да |
| L6. Согласованность с кодом | «дока не разошлась с реальностью?» | сверка с проходом C критерия №1 | да |

L1–L3 детализированы и научно обоснованы в отдельной спецификации:

> **См. [`concept-docs-readability.md`](./concept-docs-readability.md)** — формулы читаемости, prose-линтеры, инструменты со ссылками. Эта спецификация — нормативный источник для слоёв L1–L3.

L4 и L5 описаны ниже.

### L4. JTBD-присутствие (статика, без ИИ)

Для каждого из четырёх потребителей из статьи «Документация как продукт» фиксируется детерминированный чеклист: какие файлы или секции **обязаны существовать**, чтобы job в принципе мог быть выполнен. Это не оценивает качество содержимого — только его наличие. Пройти L4, не пройдя L5, можно (формальная пустышка); провалить L4 и пройти L5 — нельзя.

**Инженер-создатель** — передаёт знания без личного участия:
- `CONTRIBUTING.md` существует и непустой
- `docs/adr/` содержит ≥ 1 ADR
- `docs/intent.md` существует, одна фраза о задаче
- `docs/messages.md` содержит описание ≥ 1 доменной структуры
- `docs/contracts-graph.md` существует

**Инженер-потребитель** — за 3 минуты оценивает пригодность:
- `README.md` содержит секции (по заголовкам или явным маркерам): «что это», «интеграции» / «зависимости», «лимиты» / «ограничения», «как запустить» / «quick start»
- `README.md` ≤ ~2000 токенов в основной части (3 минуты чтения)
- `component-tests/` (или эквивалент) содержит ≥ 1 Gherkin-сценарий — исполняемая спецификация поведения
- API-разделы оформлены в pipe-формате (≤ 5 шагов)

**Менеджер** — за 3 минуты понимает структуру:
- `README.md` или `docs/architecture.md` содержит обзор системы верхнего уровня
- Перечислены slice-ы / основные подсистемы списком или таблицей
- Объём обзора ≤ ~1500 токенов

**ИИ-агент** — грузит контекст в ограниченном бюджете:
- `AGENTS.md` (или `CLAUDE.md` / `.cursorrules`) существует
- Файл содержит явные запреты («НЕ делай», «запрещено», «don't»)
- Структура — таблицы/списки, не сплошной текст (эвристика: доля строк-буллетов ≥ 30%)
- Полный бандл «факты для агента» ≤ ~8000 токенов

Эвристики наличия секций: matching по нормализованным заголовкам H1/H2/H3 против словаря синонимов. Словарь конфигурируется на проект (русский/английский, корпоративная терминология).

Выход L4: четыре чек-карточки `PASS/FAIL/PARTIAL` с перечнем отсутствующих пунктов и `file:line` для существующих.

### L5. JTBD-пригодность (симуляция через LLM)

Запускается только если L4 не упал в `FAIL`. Четыре независимых прогона, по одному на потребителя. У каждого свой job, бюджет токенов и критерий успеха:

| Потребитель | Job | Бюджет | Критерий успеха |
|---|---|---|---|
| Инженер-создатель | Сдать репозиторий преемнику без личного брифинга | `docs/` целиком | LLM отвечает на N контрольных вопросов с цитатами |
| Инженер-потребитель | За 3 минуты оценить пригодность сервиса | ~2000 токенов из README + 1 файл по выбору | LLM формулирует: что делает, какие интеграции, какие лимиты |
| Менеджер | За 3 минуты понять структуру системы | ~1500 токенов | LLM перечисляет slice-ы и их назначение |
| ИИ-агент | Загрузить контекст для конкретной задачи X | ~8000 токенов | LLM находит нужные файлы и формулирует план изменения |

Каждый прогон даёт:
- бинарный исход (job выполнен / нет),
- список конкретных пробелов («непонятно, как запустить локально», «нет описания формата сообщений», «отсутствует ADR на выбор брокера»).

### L6. Согласованность доки и кода

- `intent.md` существует и описывает задачу одной фразой (проверка длины и тональности через LLM).
- `messages.md` совпадает с реальными структурами в коде (LLM сверяет имена и поля).
- `contracts-graph.md` совпадает с графом вызовов из прохода C критерия №1 — общий источник истины двух модулей.

### Итоговая оценка по критерию №2

Четыре независимых score (по одному на JTBD) + список пробелов. **Не усредняем**: «документация для всех = ни для кого», поэтому и оценка не одна цифра. Для каждого потребителя итог = `min(L4-чеклист, L5-симуляция)` — формальное наличие без подтверждённой пригодности не считается успехом, и наоборот.

## Отчёт

Markdown + machine-readable JSON, чтобы CI мог потреблять:

```
1. Готовность по дисциплине: PASS / FAIL
2. I/O-граница:
   - жёсткие нарушения: [file:line, цитата, причина]
   - мягкие: [...]
   - граф классификации модулей
3. Документация:
   - JTBD-таблица 4×(pass/fail + пробелы)
   - структурные нарушения
   - drift по slice-ам
4. backlog.md: готовые тикеты на починку
```

Последний пункт важен: отчёт сам становится `backlog.md` в форме автора — RRA встраивается в дисциплину, а не существует параллельно ей.

## Стратегия итераций

1. Обособленный модуль документации (язык-агностичный) — даёт ценность на любом репозитории.
2. I/O-детектор для Go (язык примеров из статей).
3. I/O-детектор для TypeScript.
4. I/O-детектор для Python.
5. I/O-детектор для Java (самый сложный из-за аннотаций Spring/JPA и наследования).

## Принципы реализации

- **Один язык за итерацию.** Каталог I/O-маркеров и AST-парсер язык-специфичны.
- **Каталог разрешённых импортов** ведётся явно и пополняется по результатам ложноположительных.
- **LLM используется выборочно**, не на всём репозитории. «Opus думает, Sonnet делает»: дорогая модель — на спорные модули и JTBD-симуляцию, дешёвая — на классификацию.
- **RRA сам подчиняется правилу «остановиться и сообщить»**: при противоречии «код vs `contracts-graph.md`» это не «снижение балла», а отдельный класс отчёта «противоречие в спецификации».
