# Rational Repository Auditor (RRA) — концепция

RRA измеряет соответствие репозитория рациональной дисциплине. Не «линтер стиля» и не «отчёт о покрытии тестами». Обзор; детали слоёв и проходов — в подстраницах `docs/`.

## Источники

Концепция опирается на цикл статей по рациональной разработке:
- https://codemonsters.team/blog/2026/05/02/rational-design-skills/
- https://codemonsters.team/blog/2026/05/01/rational-design-discipline/
- https://codemonsters.team/blog/2026/04/26/component-tests-skill-passkey-demo/
- https://codemonsters.team/blog/2026/04/07/documentation-as-product-jtbd/
- https://codemonsters.team/blog/2026/04/25/testing-mythology-component-tests/

Используются формулировки автора без перевода: «программа = дерево модулей», «бизнес-логика отделена от I/O», «документация = продукт с 4 JTBD-потребителями», «качество достигается проектированием, а не тестированием».

## Цель и позиционирование

Два основных критерия:

1. **I/O отделён от бизнес-логики** — структурная корректность по построению → [`docs/criterion-1-io-boundary.md`](docs/criterion-1-io-boundary.md).
2. **Документация понятна и проста** — оценивается через JTBD четырёх потребителей → [`docs/criterion-2-docs-quality.md`](docs/criterion-2-docs-quality.md).

Прочее (TBD, формула юнит-тестов, vertical slices) — вторичные метрики, не блокирующие итоговое решение.

## Репозитории продукта

RRA поставляется как несколько репозиториев под [github.com/codemonstersteam](https://github.com/codemonstersteam/):

| Репозиторий | Роль | Slice(ы) | Статус |
|---|---|---|---|
| `rra` (этот, мета) | зонтичная концепция, оба критерия, стратегия итераций | — | актуально |
| [`rra-docs-another`](https://github.com/codemonstersteam/rra-docs-another) | критерий №2: универсальная оценка качества документации произвольных репо (L1–L6, JTBD, LLM) | `audit-docs` | ✅ пайплайн L1/L3/L4/L5/L6a в main |
| [`rra-docs`](https://github.com/codemonstersteam/rra-docs) | **гейт соответствия дисциплине** (докум. сторона): наличие/форма/согласованность артефактов + опц. LLM-тир `--semantic`. Детерминированно, для CI | — | планируется |
| [`rra-audit-repo`](https://github.com/codemonstersteam/rra-audit-repo) | критерий №1 (I/O-граница по языкам) + точка входа + отчёт; оркеструет `rra-docs-another` | `audit-repo`, `audit-iograph`, `audit-report` | планируется |

Концепция и план разработки каждого компонента — в его `CONCEPT.md` / `PLAN.md`.
`rra-audit-repo` вызывает `rra-docs-another` для критерия №2. Гейт `rra-docs` и
проход C критерия №1 делят источник истины — один `contracts-graph.md` (сверка
«дока ↔ дока» в гейте, «дока ↔ код» в `rra-audit-repo`).

## Поддерживаемые языки

Go, Java, Python, TypeScript. Каждый поддерживается отдельным «языковым модулем» с собственным каталогом I/O-маркеров и парсером AST. Один язык — одна итерация поставки.

## Архитектура самого RRA — по тем же принципам

Чтобы продукт был «правдив сам себе»:

- **Slice-и (вертикальные срезы)**: `audit-repo` (вход CLI/HTTP), `audit-iograph` (детектор I/O-границы), `audit-docs` (оценка документации, обособлен), `audit-report` (сборка отчёта).
- **I/O-модули**: `RepoStore` (git, ФС), `LLMClient` (Anthropic API), `ReportSink`. Никаких сырых `*http.Client` / `os.*` в логике.
- **Бизнес-логика**: чистые функции над AST и текстом. Юнит-тесты без моков.
- **Головной модуль**: оркеструет цепочку, принимает все решения.

## Критерий №1 — отделение I/O от бизнес-логики (кратко)

Три прохода, каждый дороже и точнее: **A** статика по AST и графу импортов (70–80% бесплатно), **B** семантика через LLM для спорных модулей (с цитатой строки), **C** сверка кода с проектным `contracts-graph.md`. Итог — структурированный результат: жёсткие нарушения (блокирующие), мягкие, покрытие графом.

Каталоги I/O-маркеров 4 языков, правила-блокеры, разрешённые импорты и промпт-чеклист — в [`docs/criterion-1-io-boundary.md`](docs/criterion-1-io-boundary.md).

## Критерий №2 — оценка документации (кратко)

Обособленный slice, запускается на любом репо без I/O-аудита. Оценка слоистая; ИИ — только на верхних слоях и только если нижние пройдены («дешёвая модель применяет, дорогая думает»).

| Слой | Вопрос | ИИ |
|---|---|---|
| L1. Читаемость | «написано ли понятно?» | нет |
| L2. Стиль | «нет ли когнитивного шума?» | нет |
| L3. Структурная полнота | «есть ли обязательные файлы и связи?» | нет |
| L4. JTBD-присутствие | «есть ли разделы под job каждого потребителя?» | нет |
| L5. JTBD-пригодность | «закроет ли содержимое job за бюджет?» | да |
| L6. Согласованность с кодом | «дока не разошлась с реальностью?» | да |

Итог — четыре независимых JTBD-score + пробелы, **не усредняем**. Детали L4–L6 — в [`docs/criterion-2-docs-quality.md`](docs/criterion-2-docs-quality.md); научная база L1–L3 — в [`concept-docs-readability.md`](concept-docs-readability.md).

## Отчёт

Markdown + machine-readable JSON для CI: готовность по дисциплине, I/O-нарушения с `file:line`, JTBD-таблица, drift, и `backlog.md` с готовыми тикетами на починку. Формат — в [`docs/report.md`](docs/report.md).

## Стратегия итераций

1. Обособленный модуль документации (язык-агностичный) — даёт ценность на любом репозитории.
2. I/O-детектор для Go (язык примеров из статей).
3. I/O-детектор для TypeScript.
4. I/O-детектор для Python.
5. I/O-детектор для Java (самый сложный из-за аннотаций Spring/JPA и наследования).

## Принципы реализации

- **Один язык за итерацию.** Каталог I/O-маркеров и AST-парсер язык-специфичны.
- **Каталог разрешённых импортов** ведётся явно и пополняется по результатам ложноположительных.
- **LLM используется выборочно**, не на всём репозитории. «Opus думает, Sonnet делает»: дорогая модель — на спорные модули и JTBD-симуляцию, дешёвая — на классификацию.
- **LLM подключается провайдер-агностично** — любая модель по OpenAI-совместимому API (дефолт — Claude нативно). См. [ADR 0001](docs/adr/0001-llm-provider-agnostic.md).
- **RRA сам подчиняется правилу «остановиться и сообщить»**: при противоречии «код vs `contracts-graph.md`» это не «снижение балла», а отдельный класс отчёта «противоречие в спецификации».
