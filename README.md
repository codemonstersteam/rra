# Rational Repository Auditor (RRA)

> Измеряет, насколько репозиторий следует рациональной дисциплине: I/O отделён от логики и документация понятна по JTBD.

Для инженеров и тимлидов, которым нужна объективная, CI-пригодная метрика дисциплины кода и доки. **Не** «линтер стиля» и **не** покрытие тестами.

## Из чего состоит

RRA поставляется как несколько репозиториев под [github.com/codemonstersteam](https://github.com/codemonstersteam/):

| Репозиторий | Роль | Статус |
|---|---|---|
| **`rra`** (этот) | зонтичная концепция, стратегия итераций, кросс-сервисные фичи | актуально |
| [`rra-docs-another`](https://github.com/codemonstersteam/rra-docs-another) | критерий №2 — оценка качества документации (L1–L6, JTBD, LLM) | ✅ в main |
| [`rra-docs`](https://github.com/codemonstersteam/rra-docs) | гейт соответствия дисциплине для CI (детерминированный) | планируется |
| [`rra-audit-repo`](https://github.com/codemonstersteam/rra-audit-repo) | критерий №1 — I/O-граница по языкам + точка входа + отчёт | планируется |

## Два критерия

- **I/O отделён от бизнес-логики** — структурная корректность по построению.
- **Документация понятна и проста** — оценка через JTBD четырёх потребителей.

## Стек

| Компонент | Технология |
|---|---|
| CLI / ядро | Go |
| Анализ кода | AST по языкам (Go → TS → Python → Java) |
| Семантика (выборочно) | LLM, провайдер-агностично (OpenAI-совместимо) |
| Линтеры доки | Vale, markdownlint |

## Как устроено (верхний уровень)

```
rra-audit-repo  ── точка входа, сборка отчёта
  ├─ критерий №1: I/O-граница ──→ AST + граф импортов
  └─ критерий №2 ─────────────→ вызывает rra-docs-another

rra-docs (гейт CI)  ──┐
rra-audit-repo, проход C ──┴─ общий источник истины: contracts-graph.md
```
