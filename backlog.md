# Backlog — Rational Repository Auditor

Верхнеуровневый список работ. Каждая позиция — самостоятельная эпопея (одна или несколько вертикальных нарезок). Внутри эпопеи раскладывается на тикеты по правилу «один тикет = один slice = одна ветка = один PR».

Порядок отражает стратегию поставки из `CONCEPT.md`: ценность раньше широты охвата.

---

## 0. Проектная документация RRA

**Цель.** RRA подчиняется собственной дисциплине: до кода — спецификация.

**Поставка.**
- `docs/intent.md` — одна фраза о задаче продукта
- `docs/messages.md` — каталог доменных структур (`Repo`, `Module`, `Violation`, `JTBDResult`, `Report`)
- `docs/contracts-graph.md` — граф вызовов между slice-ами
- `docs/infrastructure.md` — описание I/O-модулей (`RepoStore`, `LLMClient`, `ReportSink`)
- `docs/architecture.md` — устройство по slice-ам
- `docs/adr/0001-language-iteration-order.md` — почему Go → TS → Python → Java

**Готовность.** Документы согласуются между собой; LLM-симуляция «инженер-потребитель» за 3 минуты отвечает, что делает продукт.

## 1. Slice `audit-docs` — обособленный модуль оценки документации — ✅ поставлено как `rra-docs-another`

**Статус.** Реализовано отдельным репозиторием [`rra-docs-another`](https://github.com/codemonstersteam/rra-docs-another): пайплайн **L1/L3/L4/L5/L6a** в main (Option A — одна добыча входа + листья `Evaluate`). Смягчён гейт L5 (`hasDocs` вместо short-circuit; FAIL L4 кэпит L5 до PARTIAL — ADR 0004). **Открыто, опционально:** L2 `style` отложен в TBD (внешние линтеры не тянем; состав L2 проектируем от JTBD); L6c `drift --semantic` — follow-up за флагом.

**Цель.** Дать ценность на любом репозитории без поддержки конкретных языков. Шестислойная оценка по `CONCEPT.md` критерий №2; нормативные источники — `concept-docs-readability.md` (L1–L3).

**Поставка.**
- L1. Читаемость: интеграция `textstat` и формулы Оборневой для русского.
- L2. Стиль: профиль Vale + markdownlint-cli2; кастомные правила терминологии для проекта.
- L3. Структурная полнота: shell/Go-скрипт — обязательные файлы, ссылки между ними, drift, дублирование.
- L4. JTBD-присутствие (без ИИ): четыре чек-карточки `PASS/FAIL/PARTIAL` по словарю синонимов заголовков. Поддержка русского и английского.
- L5. JTBD-пригодность (LLM): четыре независимых прогона с бюджетами токенов и контрольными вопросами.
- L6. Согласованность с кодом: сверка `intent.md` / `messages.md` / `contracts-graph.md` с реальностью (общая инфраструктура с критерием №1).
- Отчёт: четыре независимых score по JTBD + список пробелов. Не усреднять.

**Готовность.** На репозитории-эталоне (например, passkey-demo из статьи) все четыре JTBD проходят L4 и L5. На «плохом» репозитории — L4 даёт конкретные пробелы с локациями ещё до запуска LLM.

## 2. Slice `audit-iograph` — детектор I/O-границы (Go)

**Цель.** Реализовать критерий №1 для языка примеров из статей.

**Поставка.**
- AST-парсер Go и граф транзитивных импортов.
- Каталог I/O-маркеров Go из `CONCEPT.md` (`net/http`, `database/sql`, `time.Now`, …).
- Каталог разрешённых импортов в логике (`errors`, `strings`, `strconv`, …).
- Проход A (статика): классификация модулей + жёсткие/мягкие нарушения с `file:line`.
- Проход B (LLM-чеклист): только для спорных модулей, с обязательной цитатой.
- Проход C: сверка кода с `contracts-graph.md`, если он есть.
- Детектор моков в юнит-тестах модулей логики (`gomock`).

**Готовность.** На репозитории-эталоне нарушений нет. На репозитории с `*sql.DB` в доменной функции — нарушение поймано с цитатой строки.

## 3. Slice `audit-iograph` — TypeScript

**Цель.** Тот же критерий для TS-стека (Node и фронт).

**Поставка.**
- AST через `@typescript-eslint/parser` или ts-morph; граф импортов с учётом path-alias.
- Каталог I/O-маркеров TS из `CONCEPT.md`, включая DOM (`window`, `localStorage`) и `fetch` в доменных функциях.
- Детектор моков: `jest.mock` с заменой зависимостей в тестах логики.
- Обработка особенностей: barrel-файлы, re-export, decorators (NestJS).

**Готовность.** Тестовая выборка из 3+ открытых TS-репозиториев классифицирована корректно; ложноположительные обсуждены и зафиксированы в каталоге исключений.

## 4. Slice `audit-iograph` — Python

**Цель.** Критерий №1 для Python-стека.

**Поставка.**
- AST через `ast` stdlib + `libcst` для сохранения позиций.
- Каталог I/O-маркеров Python (`requests`, `sqlalchemy`, `datetime.now`, …).
- Детектор моков: `unittest.mock`, `pytest-mock` в тестах модулей логики.
- Обработка особенностей: динамические импорты, monkeypatching, dataclass vs Pydantic.

**Готовность.** Тестовая выборка из 3+ Python-репозиториев классифицирована корректно.

## 5. Slice `audit-iograph` — Java

**Цель.** Критерий №1 для Java-стека. Оставлен на конец из-за сложности.

**Поставка.**
- AST через JavaParser или Spoon.
- Каталог I/O-маркеров Java + аннотационные сигналы (`@RestController`, `@Repository`, `@Autowired DataSource` в slice-логике).
- Каталог разрешённых импортов (`java.util.*` immutable, `Optional`, доменные пакеты).
- Обработка особенностей: наследование, `@Configuration`/`@Bean`, AOP-прокси, generics.

**Готовность.** Тестовая выборка из 3+ Spring Boot-репозиториев классифицирована корректно. Аннотационные нарушения отличаются от импортных в отчёте.

## 6. Slice `audit-report` — сборка отчёта

**Цель.** Единый отчёт по двум критериям, потребляемый человеком и CI.

**Поставка.**
- Markdown-отчёт со структурой из `CONCEPT.md`.
- Параллельный JSON для CI.
- Генерация `backlog.md` в форме автора — отчёт сам становится списком тикетов на починку.
- Отдельный класс отчёта «противоречие в спецификации» (код vs `contracts-graph.md`).

**Готовность.** На репозитории с реальными нарушениями отчёт читается без обращения к исходникам; CI-job падает по жёстким нарушениям.

## 7. Slice `audit-repo` — точка входа

**Цель.** Запуск аудита целиком: CLI и HTTP.

**Поставка.**
- CLI: `rra audit <path>`, `rra audit --docs-only <path>`.
- HTTP: `POST /audit` (для интеграции в pre-commit-сервисы).
- I/O-модули `RepoStore` (git clone / local path), `LLMClient` (Anthropic API), `ReportSink` (файл / stdout / S3).
- Конфигурация: каталог разрешённых импортов, переопределение I/O-маркеров на проект.

**Готовность.** `rra audit .` на репозитории RRA проходит сам себе. Интеграционный сценарий: создан слайс с нарушением → CI падает.

## 8. Эталонные репозитории и набор фикстур

**Цель.** Зафиксировать критерий «правильно/неправильно» исполняемыми примерами.

**Поставка.**
- Эталон «хорошо»: минимальный сервис на каждом из 4 языков, соответствующий дисциплине.
- Эталон «плохо»: тот же сервис с явными нарушениями (сырая `*sql.DB` в логике, `Date.now()` внутри pure-функции, моки в юнит-тестах, `@Repository` в slice-логике).
- Снэпшот-тесты RRA против обоих эталонов на каждом языке.

**Готовность.** Регрессии в детекторе ловятся снэпшотами автоматически.

## Заметки по инфраструктуре

- **CI для `rra-docs` и `rra-audit-repo`** — завести при старте реализации (сейчас в
  них только доки, Go-кода нет). Образец — `rra-docs-another/.github/workflows/ci.yml`:
  джоб `build` (gofmt / vet / go test) + джоб `component-tests` (docker compose →
  `component-tests/scripts/run-tests.sh`). Копировать, когда появятся `go.mod` и
  компонентные сценарии слайса.

## Принципы работы с этим backlog

- Каждая эпопея → разложить на тикеты по slice-ам **до начала** работы. Тикет не стартует без `intent.md` слайса.
- TBD: main всегда зелёный, ветки живут часы–день, fast-forward или squash.
- При обнаружении противоречия в спецификации **остановиться и сообщить**, а не реализовывать как-нибудь.
- RRA проверяет себя на каждом PR: жёсткие нарушения = блокировка слияния.
