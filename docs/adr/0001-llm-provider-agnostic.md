# 0001 — Провайдер-агностичный LLMClient (любая LLM по OpenAI-совместимому API)

## Статус

accepted

## Контекст

RRA использует LLM в трёх местах через автономный I/O-объект `LLMClient`:
гейт `rra-docs` (тир `--semantic`), `rra-docs-another` (L5 JTBD-симуляция, L6c),
`rra-audit-repo` (проходы B и C критерия №1).

Не должно быть привязки к одному вендору. Нужна возможность подключить **любую
LLM** — облачную или локальную/самохостящую — через де-факто стандарт,
OpenAI-совместимый API, не меняя бизнес-логику.

Дисциплина это позволяет «бесплатно»: провайдер скрыт внутри автономного
I/O-объекта, логика видит только доменный метод (`Simulate`/`Judge`/`Classify`),
поэтому смена или добавление провайдера её не трогает.

## Решение

`LLMClient` — провайдер-агностичный порт с двумя адаптерами за одним доменным
методом:

- **`anthropic`** (по умолчанию) — нативный Anthropic API, последний Claude,
  prompt caching и прочая вендор-специфика.
- **`openai`** — OpenAI Chat Completions API; через override `--llm-base-url`
  подключает **любой OpenAI-совместимый эндпоинт**: OpenAI, Azure OpenAI, шлюзы
  (OpenRouter / Together / Groq), локальные модели (Ollama / vLLM / LM Studio).
  Это и есть механизм «подключить любую LLM».

Конфигурация (одинаково во всех трёх репо):

- `--llm-provider anthropic|openai`, `--llm-base-url <url>`, `--llm-model <id>`;
- ключ — из env (`ANTHROPIC_API_KEY` / `OPENAI_API_KEY`), не в аргументах;
- переопределяется per-project через `--config`.

Ошибки **любого** провайдера маппятся в одни доменные коды:
`llm_rate_limited` / `llm_unavailable` / `llm_budget_exceeded`. Логика и отчёты
не знают, какой провайдер за портом.

Дефолт остаётся Claude (нативно) — как наиболее способная модель и ради
prompt caching; OpenAI-совместимый путь — опция для всего остального.

## Последствия

**Проще:**
- Нет лок-ина; работает с локальными/самохостящими моделями и шлюзами.
- Бизнес-логика и контракты модулей не меняются — провайдер целиком внутри
  I/O-объекта.

**Сложнее:**
- Порт держит только **переносимое подмножество**; провайдер-специфика
  (prompt caching — Anthropic; JSON/structured output — `response_format` у
  OpenAI vs tool-use у Anthropic; подсчёт токенов) живёт внутри адаптера.
- Нужны conformance-проверки каждого адаптера против стаб-эндпоинта в
  компонентных тестах.
- Качество JTBD-симуляции и семантических проверок зависит от выбранной модели —
  это ответственность оператора, не RRA.

Отражено в `CONCEPT.md` / `PLAN.md` каждого LLM-использующего репо (секция
«LLM-провайдеры», строка `LLMClient` в таблице I/O).
