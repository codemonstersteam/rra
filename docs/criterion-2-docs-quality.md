# Критерий №2 — модуль оценки документации (обособленный)

> Деталь концепции. Обзор и место критерия — в [`CONCEPT.md`](../CONCEPT.md).
> Реализация — утилита [`rra-docs-another`](https://github.com/codemonstersteam/rra-docs-another).

Отдельный slice — можно запускать поверх любого репозитория без I/O-аудита. Это согласуется с принципом «документация = продукт».

Оценка слоистая. ИИ привлекается **только на верхнем слое** и **только если** все нижележащие пройдены. Принцип: «дешёвая модель применяет, дорогая думает» — статика и линтеры дают 90% сигнала бесплатно, LLM добивает остаток.

| Слой | Отвечает на вопрос | Инструмент | ИИ |
|---|---|---|---|
| L1. Читаемость | «написано ли понятно?» | Flesch-Kincaid, Оборнева | нет |
| L2. Стиль | «нет ли когнитивного шума?» | Vale, markdownlint | нет |
| L3. Структурная полнота | «есть ли обязательные файлы и связи?» | shell/Go-скрипт | нет |
| L4. JTBD-присутствие | «есть ли разделы под job каждого потребителя?» | статический чеклист | нет |
| L5. JTBD-пригодность | «закроет ли содержимое job за бюджет?» | LLM-симуляция 4 ролей | да |
| L6. Согласованность с кодом | «дока не разошлась с реальностью?» | сверка с проходом C критерия №1 | да |

L1–L3 детализированы и научно обоснованы в отдельной спецификации:

> **См. [`concept-docs-readability.md`](../concept-docs-readability.md)** — формулы читаемости, prose-линтеры, инструменты со ссылками. Эта спецификация — нормативный источник для слоёв L1–L3.

L4 и L5 описаны ниже.

## L4. JTBD-присутствие (статика, без ИИ)

Для каждого из четырёх потребителей из статьи «Документация как продукт» фиксируется детерминированный чеклист: какие файлы или секции **обязаны существовать**, чтобы job в принципе мог быть выполнен. Это не оценивает качество содержимого — только его наличие. Пройти L4, не пройдя L5, можно (формальная пустышка); провалить L4 и пройти L5 — нельзя.

**Инженер-создатель** — передаёт знания без личного участия:
- `CONTRIBUTING.md` существует и непустой
- `docs/adr/` содержит ≥ 1 ADR
- `docs/intent.md` существует, одна фраза о задаче
- `docs/messages.md` содержит описание ≥ 1 доменной структуры
- `docs/contracts-graph.md` существует

**Инженер-потребитель** — за 3 минуты оценивает пригодность:
- `README.md` содержит секции (по заголовкам или явным маркерам): «что это», «интеграции» / «зависимости», «лимиты» / «ограничения», «как запустить» / «quick start»
- `README.md` ≤ ~2000 токенов в основной части (3 минуты чтения)
- `component-tests/` (или эквивалент) содержит ≥ 1 Gherkin-сценарий — исполняемая спецификация поведения
- API-разделы оформлены в pipe-формате (≤ 5 шагов)

**Менеджер** — за 3 минуты понимает структуру:
- `README.md` или `docs/architecture.md` содержит обзор системы верхнего уровня
- Перечислены slice-ы / основные подсистемы списком или таблицей
- Объём обзора ≤ ~1500 токенов

**ИИ-агент** — грузит контекст в ограниченном бюджете:
- `AGENTS.md` (или `CLAUDE.md` / `.cursorrules`) существует
- Файл содержит явные запреты («НЕ делай», «запрещено», «don't»)
- Структура — таблицы/списки, не сплошной текст (эвристика: доля строк-буллетов ≥ 30%)
- Полный бандл «факты для агента» ≤ ~8000 токенов

Эвристики наличия секций: matching по нормализованным заголовкам H1/H2/H3 против словаря синонимов. Словарь конфигурируется на проект (русский/английский, корпоративная терминология).

Выход L4: четыре чек-карточки `PASS/FAIL/PARTIAL` с перечнем отсутствующих пунктов и `file:line` для существующих.

## L5. JTBD-пригодность (симуляция через LLM)

Запускается, если документация присутствует (статика её качества не блокирует ИИ-прогон, а лишь ограничивает итог — см. L4). Четыре независимых прогона, по одному на потребителя. У каждого свой job, бюджет токенов и критерий успеха:

| Потребитель | Job | Бюджет | Критерий успеха |
|---|---|---|---|
| Инженер-создатель | Сдать репозиторий преемнику без личного брифинга | `docs/` целиком | LLM отвечает на N контрольных вопросов с цитатами |
| Инженер-потребитель | За 3 минуты оценить пригодность сервиса | ~2000 токенов из README + 1 файл по выбору | LLM формулирует: что делает, какие интеграции, какие лимиты |
| Менеджер | За 3 минуты понять структуру системы | ~1500 токенов | LLM перечисляет slice-ы и их назначение |
| ИИ-агент | Загрузить контекст для конкретной задачи X | ~8000 токенов | LLM находит нужные файлы и формулирует план изменения |

Каждый прогон даёт:
- бинарный исход (job выполнен / нет),
- список конкретных пробелов («непонятно, как запустить локально», «нет описания формата сообщений», «отсутствует ADR на выбор брокера»).

> **Смягчение гейта L5** (реализовано в `rra-docs-another`, ADR 0004): L5 запускается при наличии документации, а не только когда вся статика L4 прошла; `FAIL` на L4 не пропускает L5, а ограничивает его итог сверху до `PARTIAL`. Цель — прогон на реальных проектах и сбор статистики L4/L5.

## L6. Согласованность доки и кода

- `intent.md` существует и описывает задачу одной фразой (проверка длины и тональности через LLM).
- `messages.md` совпадает с реальными структурами в коде (LLM сверяет имена и поля).
- `contracts-graph.md` совпадает с графом вызовов из прохода C критерия №1 — общий источник истины двух модулей.

## Итоговая оценка по критерию №2

Четыре независимых score (по одному на JTBD) + список пробелов. **Не усредняем**: «документация для всех = ни для кого», поэтому и оценка не одна цифра. Для каждого потребителя итог = `min(L4-чеклист, L5-симуляция)` — формальное наличие без подтверждённой пригодности не считается успехом, и наоборот.
